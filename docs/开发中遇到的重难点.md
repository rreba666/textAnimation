# 开发中遇到的重难点

## 一、多效果组合渲染（最核心难题）

### 问题描述

预设模板需要将多个效果（如 Glitch 故障 + Neon 霓虹 + Scanline 扫描线）同时作用于同一段文字。最初尝试的方案全部失败，经历三轮重构才找到可行方案。

### 尝试一：CSS 组件绝对定位层叠

将每个效果组件用 `absolute inset-0` 层叠在预览区，所有组件渲染同一段文字。

**失败原因**：每个效果组件渲染了不透明的文字（如 Glitch 显示白色 "CYBERPUNK"，Neon 也显示白色 "CYBERPUNK"），上层文字完全覆盖下层，只能看到最顶层效果。

### 尝试二：Canvas 离屏绘制合成

将所有效果在同一块 Canvas 上按顺序绘制，每个效果贡献一层视觉。

**失败原因**：Canvas `fillText` 画不透明文字，后画的效果覆盖前面效果。即使分"辉光层→文字层→装饰层"三阶段绘制，辉光层用 `fillStyle: rgba(0,0,0,0)` 透明文字 + `shadowBlur` 发光的方式，不同效果的辉光仍然互相覆盖叠加，导致整体模糊。

### 尝试三（最终方案）：独立图层 + CSS 文字透明 + 全屏叠加

**核心思路**：利用 CSS 的 `color: transparent !important; -webkit-text-fill-color: transparent !important` 使辉光层的文字不可见，但 text-shadow 辉光和 background-clip 纹理保留。同时只让组合数组中最后一个效果作为"文字层"渲染实体文字。

**层级体系**：

| 层级 | CSS | 包含效果 | 行为 |
|------|-----|---------|------|
| glow | `color:transparent; opacity:0.35` | Neon/GlowPulse/Fire/Ice/Sparkle | 文字透明，辉光低透明度叠加 |
| deco | `pointer-events:none; opacity:0.5` | 纯 Canvas 装饰 | 背景效果 |
| text | 正常渲染 | 最后一个效果 | 实体文字清晰可见 |
| scanline | `mix-blend-mode:screen` | 全屏叠加层 | 独立于图层体系 |

**关键 CSS 代码**：
```css
.combo-glow * {
  color: transparent !important;
  -webkit-text-fill-color: transparent !important;
}
.combo-glow { opacity: 0.35; }
```

**为什么有效**：
- `color: transparent` 使文字本身不可见
- `-webkit-text-fill-color: transparent` 确保 WebKit 内核下文字填充透明
- 但 text-shadow 绘制在文字轮廓之外，不受 `color` 影响，辉光依然可见
- `opacity: 0.35` 降低辉光强度，不会遮盖上层文字
- 只有最后一个组件渲染不透明文字，保证视觉效果清晰

## 二、扫描线效果在组合中不可见

### 问题描述

扫描线（Scanline）效果在组合模式中始终不可见，无论放在哪个层级。

### 排查过程

扫描线效果使用 `background-clip: text` 技法：文字颜色设为 `transparent`，`background-image` 设为 `repeating-linear-gradient` 条纹图案，通过 `background-clip: text` 将条纹裁剪到文字轮廓内。

在组合 glow 层中，`.combo-glow * { color: transparent !important; -webkit-text-fill-color: transparent !important }` 会使文字填充透明——理论上不应对 `background-clip: text` 产生负面影响，因为 `background-clip: text` 本来就需要文字透明。

但实际操作中，`background-clip: text` 在多层绝对定位 + `isolation: isolate` + `mix-blend-mode` 的复杂环境下渲染不稳定，条纹模式在文字透明后直接消失。

### 最终方案

放弃通过 `background-clip: text` 在组件内渲染扫描线。改为在 ComboRenderer 最外层单独创建一个**全屏扫描线叠加层**：

```tsx
<div style={{
  backgroundImage: `repeating-linear-gradient(0deg, transparent, transparent ${h-1}px, ${color}33 ${h}px)`,
  mixBlendMode: 'screen',
  zIndex: 300,
  pointerEvents: 'none',
}} />
```

**优势**：
- 不依赖任何组件的文字渲染，直接覆盖在最终合成画面上
- `mix-blend-mode: screen` 使暗条纹透出下层内容，亮条纹添加 CRT 荧光感
- 动画通过 CSS `@keyframes background-position` 位移实现流动
- zIndex: 300 保证扫描线永远是画面最上层

## 三、Sparkle 星光组件导致的全局颜色污染

### 问题描述

魔法少女、霓虹都市、冰霜城堡、极地风暴四个预设的字体颜色全部显示为金橙色，与各自的粉色/蓝色/白色主题完全不匹配。

### 排查过程

逐一检查四个预设的共同点：它们都包含 Sparkle 星光效果。

查看 Sparkle 组件代码：
```tsx
<span style={{ color: '#e8c060' }}> {/* 硬编码金色！ */}
  {text || '星光文字'}
</span>
```

根因：Sparkle 组件内文字颜色硬编码为 `#e8c060`（金橙色）。在组合模式中，Sparkle 最初被归为 `deco` 层（装饰层），deco 层不会将文字透明化。因此 Sparkle 组件的金色文字作为可见文字渲染，覆盖了其他层的主题色文字。

### 修复

将 Sparkle 从 `deco` 移至 `glow` 层。`.combo-glow * { color: transparent }` 使金色文字不可见，但 Sparkle 的星点（绝对定位的 `<span>` 元素，使用 `background: #fff` + `boxShadow` 绘制）不受影响，仍然正常闪烁。

**教训**：分类层级时，不仅要考虑效果的视觉效果类型，还要检查组件内部是否渲染了硬编码颜色的文字元素。

## 四、组合中多个文字效果的双层幽灵字

### 问题描述

黑客帝国、复古电视、幽灵荧光等预设出现文字重叠——同一段文字有两个略有偏移的副本同时可见，像幽灵重影。

### 排查过程

这些预设的组合数组中包含多个"文字型"效果。例如黑客帝国：`[neon, scanline, shake, phosphor]`，其中 shake 和 phosphor 都渲染实体文字。

早期分类逻辑是按效果类型固定归类：
- `glow`: neon, glowpulse, fire, ice
- `text`: glitch, liquid, shake, phosphor（全部为 text 层）

结果 shake 和 phosphor 都在 text 层，分别渲染文字。shake 组件用 `transform: translate(x,y)` 偏移文字位置，phosphor 用 `translateX` 偏移拖尾——两个偏移量不同步，导致文字重叠出现幽灵重影。

### 最终方案

改为**按数组位置动态分类**，而不是按效果类型固定分类：

```typescript
const layers = others.map((c, i) => ({
  ...c,
  kind: i === others.length - 1 ? 'text' : (isDecoEffect(c.effect) ? 'deco' : 'glow'),
}))
```

**核心规则：只有组合数组中最后一个效果作为 text 层**，前面所有效果无论原本是什么类型，全部归为 glow 或 deco 层。这保证了任何时候都只有一段实体文字可见。

## 五、TypeScript 严格模式下的开发约束

### noUnusedLocals / noUnusedParameters

项目配置中 `noUnusedLocals: true` 和 `noUnusedParameters: true` 导致任何未使用变量都会构建失败。这在快速迭代 62 个效果组件时产生了大量类似错误：

```
error TS6133: 'i' is declared but its value is never read.
```

**解决策略**：
- 未使用的数组索引参数用 `Array.from(text).map((char) =>` 替代 `(char, i) =>`
- 未使用的函数参数用 `_paramName` 前缀（但 TypeScript 6 不完全支持，最终需移除或实际使用）

### verbatimModuleSyntax

`verbatimModuleSyntax: true` 要求 type 导入必须使用 `import type { ... }` 语法，不能混用。在导入 `EffectType` 等纯类型时必须显式标注 `type`。

### erasableSyntaxOnly

`erasableSyntaxOnly: true` 禁止使用 `enum`、`namespace` 等运行时语法，所有枚举值必须用字符串联合类型替代：
```typescript
// 不允许
enum Category { Text, Background }
// 必须
type Category = 'text' | 'background'
```

## 六、Canvas 效果组件的 React 生命周期管理

### 问题

Canvas 效果（ParticleField、MatrixRain、Noise 等）需要在组件挂载时初始化 Canvas、启动 `requestAnimationFrame` 循环，并在参数变化或组件卸载时正确销毁。

### 模式

```typescript
const render = useCallback(() => {
  const canvas = canvasRef.current!
  const ctx = canvas.getContext('2d')!
  // 设置尺寸
  canvas.width = parent.clientWidth
  canvas.height = parent.clientHeight
  
  // 监听 resize 事件重新设置尺寸
  window.addEventListener('resize', resize)
  
  function frame() {
    // 绘制逻辑
    rafRef.current = requestAnimationFrame(frame)
  }
  frame()
  
  // 返回清理函数
  return () => {
    cancelAnimationFrame(rafRef.current)
    window.removeEventListener('resize', resize)
  }
}, [dependencies])

useEffect(() => {
  const cleanup = render()
  return () => { cleanup?.() }
}, [render])
```

**关键点**：
- `useCallback` 缓存渲染函数，仅在依赖变化时重建
- `useEffect` 中调用渲染函数，返回清理函数停止 RAF 和移除事件监听
- Canvas 尺寸从 `parentElement.clientWidth/Height` 获取，保持自适应

## 七、构建产物与 Node 16 服务器兼容

### 需求

服务器端 Node 版本为 16，需要确保构建产物能在该环境下正常提供静态文件服务。

### 方案

- Vite 构建目标设为 `es2020`：`build: { target: 'es2020' }`
- 禁止使用 Node 18+ 专有 API（如 `fetch`、`structuredClone` 等不带 polyfill）
- 输出纯静态文件（HTML + CSS + JS），不依赖服务端运行时
- 打包命令：`tar -czf deploy-frontend-YYYYMMDD.tar.gz -C project dist`
